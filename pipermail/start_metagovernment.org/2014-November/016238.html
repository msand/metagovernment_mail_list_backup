A standard format would seem to me to be a requirement.  Some sort of
XML would seem appropriate.  Now are people going to have to learn
this format and mark up their text properly (and then run an app to
check it), or will there be apps that create the data in the correct
format?  Who writes these apps, and are they tied to the data hosting
service, or independent?

>>> Would you like to try it ? It would be very helpful for recognizing our
>>> opinions. You can also reuse the data in other projects. It is all
>>> decentralized......

>> I think I'm actually opposed to the idea of a totally decentralized
>> (e.g., peer to peer) system.  Too easy to spoof, and more importantly
>> too hard to detect when an alternate source for "your" data somehow
>> makes it into the chain.  Even worse, it could come and go, so even if
>> you happen to check periodically to verify what the system thinks is
>> your data it wouldn't prevent someone from changing the path a few
>> milliseconds before a "vote", then change it back immediately
>> afterward.  So, even though it basically *invites* attack, I believe a
>> centralized data store is required.  Distributed, of course, to
>> prevent single-point-of-failure, but all mirrors rather than
>> independent sources.  I'd propose some sort of versioning to allow
>> detection of hacks.  My philosophy on all of these things is that it's
>> vastly more important to be able to have a totally reliable way to
>> *detect* this sort of "voter fraud" than it is to try to prevent every
>> possible cause of it.  You can always redo a vote if there is credible
>> evidence that it was invalid, but if normal citizens can't even detect
>> tampering the government will effectively be run by sysadmins and
>> hackers.
>
> How could you know this ? I will not be peer to peer probably, I expect
> server to server. IMHO you have quick judgement.

What you're describing *is* peer to peer (i.e., if a central authority
doesn't control the servers, it's P2P, even if the peers are servers).
And this is not something I couldn't know, or that requires quick
judgement: I've been dealing with UNIX and networked systems for 35
years now, and yet am still regularly surprised (and disappointed) by
how often (and how easily) they get hacked.  Which leads to my expert
opinion being that a system that relies exclusively on security to
*prevent* hacking is unworkable, as is one that lacks a reliable way
for people to check the outputs.  Still, we ought to design it to be
as hack-proof as possible, if only to improve efficiency, which means
P2P is probably out as an architecture unless there is some sudden
breakthrough in that technology in the next few years...
