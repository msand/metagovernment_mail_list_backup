 We DO use Email as the main functionality of our application.
(Where our current application is this discussion on the metagovernment

-(pro 50%)Modern email applications allow for advanced functionality
in the email.
>
> => I guess they do.


>
> --(con 100%)Almost all email readers limit HTML features, either for security reasons, or because of the nature of the interface. CSS and JavaScript can be especially problematic. (http://en.wikipedia.org/wiki/Comparison_of_e-mail_clients)
>
>
=> true, thats what makes it a more challanging task, but it has to start
somewhere, thats why i suggested to ask people on this list which email
client they use. (i do use google mail)
Then watch out for the "least common denominator" on what those clients do
support.
And one could start with the most common email clients.


>
> --(pro 100%)Indeed many email clients allow for JavaScript, HTML, and CSS
>
> => thats an enabler. If they wouldnt, we could not do anything to improve
it.


> -(pro 50%)Most people use email most of the time
>
> => I guess, and its one thing most people complain about, that answering
emails takes a long time in their daily routine. ...enhancing email with
whats possible through html/javascript/css will actually create value if it
makes it easier to work through emails


> --(pro 100%)Email is used significantly as a source of information and communication
>
> => yes. there could be a world without facebook and without twitter, that
would not break the economy, but if email were removed, people could not
get their passwords, couldnt register for new plattforms, couldnt
communicate in the business world, many technical services use email too,
...that would just break a lot, so thats why i believe email is significant.



> --(con 100%)92% people read their email online. 19% of the person's time online is spent on emails. The other 4/5th of the time they are using web applications (http://www.go-gulf.com/blog/online-time/)
>
> => Yes, thats what i was thinking. So 1/5 of total time spent online is
EMAIL :-)
And the thing thats important to notice is... the other 4/5 might be spent
using web applications and other stuff, BUT: for most people, the web
applications they use are different, so i believe, beside facebook or
google, there might be no web apps, which are used by ALL people in a
similar frequency or duration.
How those 4/5 are put together is very different for each person, but 1/5
email is more or less true for everyone, which means, its a common
technology which ALL people use 1/5 of their time spent online. Thats a
huge advantage.



> -(con 100%)Email can be used as a rich notification system using all it's features without having to be used as the main interface for an service.
>
> => I do not believe it needs to be "the main" interface for a service, but
i believe it should start with email (like twitter started with its
website) but now has an API and many "twitter client apps" too, which allow
more advanced usage.
"Email Clients" just should be the first interface and a basic one that
improves what we have now, without the need for people to install
additional software... they receive the functionality or app with the email
and its content :-)
If people start using those features (if WE start to use this features to
improve the efficiency of our metagovernment proposal/discussion process on
this mailing list), we could evaluate and improve on what people need,
while we discuss, because we use it while we discuss.
At some point, we could come up with an API too and enable others to create
"stand alone clients" which can communicate with the same plattform, but
not having to use email.





>
> -(con 100%)Email is a snapshot in time and does not always show the latest information which can cause confusion.
>
> => If email is enhanced by html/css/javascript, you could "add" what you
say is missing :-)
The pirate party once in a while sends an email with statistics about usage
of the mailinglist.
Many mailing lists apply footers to all their emails by default, ...so why
wouldnt you be able to add some kind of "dashboard" or "cockpit view" to
all emails to provide the context that normal emails lack.



> -(con 100%)Email is not a trusted identity system. An moderately technical person can spoof another person's email address.
>
> Who cares :-) It has to start somewhere and if it really gains traction,
those "stand alone clients" could be offered or gpg encryption or something
could be added to at least "sign" emails. This is a generic argument which
i do not find very helpful/constructive :-) Thats a problem which can be
solved somehow if it really becomes important in the future because people
really RELY on the results, but until we get there, its just not important.


-(con 100%)Many people turn off HTML email for security reasons
>
> Same argument. I do believe, that most people just DONT turn it off.
The only people i know which turn it off are IT people itself, but most
people dont know that its possible to turn it off, nor do they know how to
turn it off, nor would they do it :-)
