Regarding spam on the wiki. See
http://stackoverflow.com/questions/6748633/how-to-prevent-mediawiki-spam

Mike,

Votorala seems to be a promising platform. The good documentation is
impressive.

Well, trust declines when information is hidden.  Nothing beats full
> disclosure - all the cards on the table.  Anonymous voting (private
> voting) makes the results *comparatively* less credible.
> More important are the social aspects of voting.  We don't easily
> trust people who hide behind masks.  Leaders will tend to vote in
> public (non-anonymous) and others will tend to follow.  So the bulk of
> initial development resources (which are scarce) must go to public
> voting, or else the users will go to the competition.


I agree with what you say. If you want to focus on public
voting that's fine but I think the platform should be built in such a way
to eventually allow private voting. Is Votorola built this way?

> Hi Ed,
>
> > Michael, did I get that right ... ?
>
> I think only admins can edit that blacklist.  So only they can add
> links just now, and only by fiddling with the blacklist.
>
> > ... and... any ideas for a less drastic fix?
>
> A possible stopgap (not a fix): Someone volunteers to help me patrol
> the recent changes at least once a day.  It's tedious and depressing
> work, so this can only be a stopgap.
>
> (1) A possible longer term fix:
> http://www.mediawiki.org/wiki/Extension:NotEvil#Cleaned-up_code
>   * New and anonymous users cannot post links.
>   * There's a new group of "trusted" users.  Only they (and admins and
>     such) can post links.
>   * Admin adds new users to "trusted" group when they prove themselves
>     human, or when they request it.  (So this is not a permanent fix
>     if we start to grow.)
> I'd need to test this on the pollwiki first.  If it works as promised,
> then it might enable us to drop some (maybe all?) the other defences.
>
> (2) Or cooperate with other victims:
>
> http://www.mediawiki.org/wiki/Anti-spam_features#Bad_Behavior_and_Project_HoneyPot
> This looks more difficult to set up, but seems the right way to go.
>
> (3) If all else fails, we could add a custom hurdle to fool spambots.
> E.g. all edit comments must begin with 'notspam' or some such token,
> which we supply pre-written along with an explanation, then strip away
> internally.  But that's kind of intrusive.
>
> Should I look into (1) or (2)?
>
