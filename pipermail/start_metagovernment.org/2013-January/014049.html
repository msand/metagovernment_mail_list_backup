Hi Zohar,

I second what Ed says.  You're welcome to run with a piece of Votorola
or Outcast if you like.  Otherwise, I hope we can still cooperate in
some way.  I think cooperation is the future for this field.

Zohar Jackson said:
> Votorala seems to be a promising platform. The good documentation is
> impressive.

(Thank you, we still need to document the practice.  It's been hard to
foresee, and we're only starting to come to grips with it now.)

> I agree with what you say. If you want to focus on public voting
> that's fine but I think the platform should be built in such a way
> to eventually allow private voting. Is Votorola built this way?

Yes, I think so.  Conseo alludes to a simple solution we tried to
hammer out once: http://metagovernment.org/wiki/Streetwiki#Proxy_blind
It has problems, as you see.  Maybe they can be fixed.  In any case,
we have full control (in the code) over how we identify private voters
in public interfaces.  If we didn't care about public verification,
then we could simply black out the real identifier and turn the vote
into a secret ballot.  But we do care, so we must replace the real
identifier with a registry identifier (I think that will always be the
case) that nevertheless allows for some form of public verification.
So the voting part is fairly easy.

The real work is in the registry.  But we can support any number of
registries or (like proxy blind) registry extensions.  That's a design
requirement.  So I think it means we can support any number of private
voting schemes.  Does this make sense?


Conseo, as I understand, pipes are unconnected with private voting.
You're right to suggest that a private voter *as such* ought not to
communicate with anyone.  It comprises secrecy.  (Of course, such a
restriction defeats the main purpose of using Votorola in the first
place, as it's mostly about communication.  But if someone wants to
vote and can only vote anonymously, then it seems wrong to say no.)

Mike
