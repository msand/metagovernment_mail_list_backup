Hello,

Ed Pastore wrote:
> For example, a small nonprofit wishing to organize themselves using
> our softwares might want to use Candiwi to draft their bylaws,
> Vilfredo to develop ideas, and Votorola to make formal decisions. They
> would benefit greatly from having user portability between the engines
> (including a trust metric and user-tags)... 

Yes, I like this use-case very much. And I too see the great benefit of 
the user portability.
If the others here agree too, that is something we could start working 
on right away - together. Mike is making good progress with the 
trustservers and streetwikis. And here in Germany there is also 
consensus about this with two other e-dem-projects (Adhocracy and Liquid 
Feedback).

> I think where Pietro and I are both getting
> lost is in the need for having the same issue voted on by multiple
> engines.
>
> That may be necessary in large-scale implementations, but if we take
> the more patient approach of trying to get adopted first in very small
> communities, then it becomes unnecessary to allow multiple voting
> systems on the same issue (doesn't it?).

Yes, I mean large-scale implementations.
Ed, here in Germany the whole e-dem-thing is really boiling. The 
Pirate-Party (12000 members only in Germany) is desperatly looking for a 
running software to make its internal desicions with it (and to later 
open it up to the whole public). Their only problem with it right now 
is, that there are 4-5 half-ready ones to choose from. I liked your 
patient approach with the chess-club-example very much, but at least 
here in Germany this patience is not necessary anymore.

In such a large-scale use-case it seems to make at least some sense, to 
let each user choose, which software to use. Especially because there 
are actually really different ones, which still address the same 
use-cases. Votorola and Candiwi are very much alike compared to Liquid 
Feedback for example. But Votorola and Liquid Feedback have unlike 
Vilfredo exactly the same use-cases.
And also I dont think, there is a way to prevent users to start voting 
on the same issue, but using a different tool. If just in the case, that 
they dont know of each other yet, voting on some global issue.

Pietro wrote:
> One thing is to try to mirror everything from one to the other. I 
> think that's
> doomed to fail because each project is different.

If you mean by "everything", that no information is lost - then you are 
right. Some information will be lost in some translations (not in all). 
But why not take, what is possible? It would be like giving up to 
translate from Eskimos, just because they have more words for "snow" 
than we have. There is true information coming through in translating 
+10 to a "Yes", isnt it?

Greetings,
Thomas
