Jonas Liljegren wrote:
> 2011-09-21 23:57, David Gleason skrev:
> > A few random thoughts:
> > 
> > - I agree that collaborative governance has to start from the bottom up.
> > 
> > - c.g. has to be based on open source software,and must be provable as
> > mathematically correct, i.e. as simple as possible (see some of Alan
> > Kay's recent work e.g.
> > http://www.tele-task.de/de/archive/video/flash/14029/), and inherently
> > bug-free.
> > 
> > - The system must deal with sets of simple issues at a granular level,
> > i.e. things that can be resolved to a single change in a process or
> > resource allocation.  These could be grouped into larger things but the
> > granularity has to be there to allow the full extent of any proposal to
> > be completely clear, with references/sources provided, etc.  That way
> > people can refine and comment on individual things as needed, but it
> > won't turn into a huge blob of text and subjective interpretations.
> > 
> > - A network-oriented architecture must be taken in the software, which
> > manages complexity by modeling things as networks of nodes (resources,
> > people, groups, organizations) and dynamic relationships.  In other
> > words it has to model the real world.  (Kind of like object-oriented
> > programming, but without the emphasis on programming, without the
> > limitations of typical OO structures, and with an equal emphasis on the
> > dynamic relationships between objects.)  If a system does not accurately
> > capture existing reality, most people will not be able to understand the
> > existing reality, what changes or optimizations are therefore
> > appropriate, or what effects those changes would have.
> > 
> > (I see this above point as the biggest issue, and am now developing a
> > web information system that makes it easy for non-programmers to
> > construct and share any type of data structures and relations to model
> > those kinds of things.  It's probably many ~6 months away from being
> > useful however.)
