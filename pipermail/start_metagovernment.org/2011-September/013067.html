A few more comments on the subject of a new type of network-oriented information system:
- The data model has to be simple.  The concept of XML/RDF/OWL/semantic web "3.0" has gone in the wrong direction and these are too complex and error prone for use by normal people.  They are probably great for enterprise-level high-budget projects where data will be used more by machines than by people, but they are too complex to be realistically usable by the general public and less experienced developers.  I'm creating something that will have the same flexibility, few limitations, and won't require 1000's pages of specs to define.  My initial target applications are not collaborative governance, but simpler things like CMS's, since thos are what everyone is using and they are easy to extend so I see that as the most obvious place to move information itself from being static and isolated to being dynamic and interconnected.
- The GUI is a fairly minor detail for me, compared to first addressing the complexity barrier that now causes most development to be relatively slow, bug prone, and isolated, due to too many technologies and too much inconsistency.  Collaborative governancedoes not have a big enough budget behind it for it to be possible to implement a good enough system by using typical current technology approaches.  I encourage everyone to watch this recent presentation by Alan Kay to get an idea of what I mean http://www.tele-task.de/de/archive/video/flash/14029/  (Alan Kay was the guy who first coined the term "object-oriented", yet he has a very different definition of that than what is has more recently come to imply in programming languages like C++, Java, etc.  His current STEPS project has been able to take the typical 20M lines of code associated with a typical PC SW stack and reduce it in size by 1000 times. This is perhaps the most significant computer
 science project so far of the century.)
- Currently I'm just using PHP and MySQL (with InnoDB engine).  At some point I might have to go with a database that more directly accommodates OO or graph-oriented data but mysql is working fine at the moment.  Fortunately I'm not trying to model anything complicated now but am just focusing on basic CMS types of applications.

> 2011-09-21 23:57, David Gleason skrev:
>> A few random thoughts:
>>
>> - I agree that collaborative governance has to start from the bottom up.
>>
>> - c.g. has to be based on open source software,and must be provable as
>> mathematically correct, i.e. as simple as possible (see some of Alan
>> Kay's recent work e.g.
>> http://www.tele-task.de/de/archive/video/flash/14029/), and inherently
>> bug-free.
>>
>> - The system must deal with sets of simple issues at a granular level,
>> i.e. things that can be resolved to a single change in a process or
>> resource allocation.  These could be grouped into larger things but the
>> granularity has to be there to allow the full extent of any proposal to
>> be completely clear, with references/sources provided, etc.  That way
>> people can refine and comment on individual things as needed, but it
>> won't turn into a huge blob of text and subjective interpretations.
>>
>> - A network-oriented architecture must be taken in the software, which
>> manages complexity by modeling things as networks of nodes (resources,
>> people, groups, organizations) and dynamic relationships.  In other
>> words it has to model the real world.  (Kind of like object-oriented
>> programming, but without the emphasis on programming, without the
>> limitations of typical OO structures, and with an equal emphasis on the
>> dynamic relationships between objects.)  If a system does not accurately
>> capture existing reality, most people will not be able to understand the
>> existing reality, what changes or optimizations are therefore
>> appropriate, or what effects those changes would have.
>>
>> (I see this above point as the biggest issue, and am now developing a
>> web information system that makes it easy for non-programmers to
>> construct and share any type of data structures and relations to model
>> those kinds of things.  It's probably many ~6 months away from being
>> useful however.)
