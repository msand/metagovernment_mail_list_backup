It is quite similar.

There is a demo of my current state at
http://test.liqd.net/page/Section_2__The_House (a multi-variant wiki
page) and http://test.liqd.net/page/diff?left=14&right=15 (a
rudimentary diff, this should really only show the passages that have
changed at all). I still need to fully resolve the issue of which
comments to show where (i.e. will users realize that there is supposed
to be a discussion space between documents? how can I allow distinct
discussions on page variants without fatally confusing people?).

The whole idea of this is to have a HEAD variant ("status quo") of
each legal document and to allow users to create variants of that
version (or to create a new document with an empty HEAD).

This is a second step after formulating the goal of the change (e.g.
"better benefits for people on health care") as a proposal. Variants
regarding a number of documents can then be voted to become the "body"
(i.e. the implementation section) of that proposal. Finally (pending
some consistency checks), the proposal (not the variants) can be
decided upon.

I don't use a traditional wiki engine for this because I want to
tightly integrate with the (delegated) voting aspect of the system.
IMHO this kind of deep-down policy text work is where DV will become
especially powerful.

There is a very rough design document at
http://piratepad.net/adhcplanbeta. (This has partially been
implemented and parts of it are still on my roadmap. The whole
discussion on whether to use an existing DVCS was never really
finished and I decided to roll my own, mostly because I would not have
been using more than 5% of any DVCS while having to handle all of its
complexity. I have not yet gotten into the whole geo-debate which
seems very central to the pollwiki concept, I'm still waiting for a
"customer" to give me proper requirements on this ;))

Yet you won't get over the requirement of being able to extract some
