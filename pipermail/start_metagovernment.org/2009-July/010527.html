Then I guess the next step is to actually do it.  It probably starts
like this (correct me if I'm wrong):

  The technical details of each interface are to be specified by the
  developers, then posted for critique by others. The development
  process can be summarized as "rough consensus and running code". It
  will take two developers - one on the server side, and one on the
  client side - to get the process rolling for any particular
  interface.

  http://metagovernment.org/wiki/Standardization#Public_Interfaces

So it takes 2 developers and 2 tools to bootstrap it.  The developers
work together, and soon their tools are working together.

For my part, I have Votorola running online.  I'm currently working to
interface it with external drafting tools (such as wikis) and Web
sites (such as blogs).  Maybe it will be the first demo of the open
architecture in action - unless Jim Gilliam beats me with his
interface to Facebook, or FP7 gets its code running online.

On the other hand, the real test is to get two *e-dem* tools from
*separate* projects working together.  If we can do that - and do it
without alienating other projects and other developers, but rather
inviting them - then we're going to be unbeatable.  FP7 will win its
next request for funding, Metagov will be center stage, and
e-democracy will take a step forward.

Is this the right way to proceed?  If so, I'm open to interface
proposals, and I'll soon be making proposals of my own.
