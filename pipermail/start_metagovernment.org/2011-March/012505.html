Hi Ed,

I don't know that I'm going to be able to review Votorola's documentation
any time soon.  I need to get CrowdVote to at least a proof-of-concept stage
in time for a ColdFusion conference in May.

I think we are going to continue along our current path only because four
other developers are already working with me at this point and a big
side-objective of ours is just getting out a solid ColdFusion application
running on the Open Source CFML Server http://openbluedragon.org/ and
deployed for free on Google's App Engine.  We think this deployment model is
going to get a lot of attention in the Gov 2.0 community, so we're eager to
get a few examples out there.

I think we would be honored to have it listed on your Active Projects page.


Heh all I meant by "heady stuff" is I just saw some quotes from Habermas and
Gadamer on the Votorola Theory page and thought: "boy these guys are
serious!"  But these are difficult issues so I can understand the line you
have to walk between making it approachable but not watering things down.

Also Michael asks in the next email but I'll answer here - yes CrowdVote
definitely will have a fully REST-ful API at some point.  We want CrowdVote
to be more of a distributed platform than a service from any one provider.

BTW, has anyone seen https://www.votizen.com/about/ ?  Really interesting
stuff.  It lacks CrowdVote's "social representation market," and it's tied
to one Docket (State or National offices).  But it does a fascinating job
with looking users up on Voter Registration Lists.

Between your work and sites like https://popvox.com/ and http://votizen.com,
momentum really seems to be gathering towards a radical re-thinking of
representative democracy.
