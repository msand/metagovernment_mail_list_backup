>
>  So maybe if we take Kris's summary points, and we view them against
>> what Thomas is asking for?  Something like this:
>>
>>  1. 'Stakeholder involvement' rather than anyone's (outsider's)
>>    involvement.
>>
>>    The problem here, I guess, is that the projects are split on
>>    whether or not to enforce this?
>>
>>    But there's a technical solution to that, right?  The user can
>>    filter out the votes at a higher level according to the criterion
>>    of stakeholder involvement - or not filter them.  So we can have
>>    it both ways.
>>
>
> By user, do you mean admin?
>
> In any event, the idea of stakeholdership does not have to be strictly
> enforced across the board. All we need is some common standard that can
> place the user geographically, such as a postal code, perhaps?
>

To me, user and stakeholder is the community member. Admin should be a open
system of coordinators for the tool functioning.

The user can be defined by the community and filters set. For example, I own
a piece of land in my home town where drip irrigation is done. But I am
presently employed in the city. I should be eligible to vote on the land
issue.

>
>
>
>   2. Unique identity of the stakeholder / voter.
>>
>>    Ed is warning us that this could slow up the deployment, or
>>    restrict its reach.  But again, there's maybe a technical
>>    solution: what if the authentication facility has an on/off
>>    switch?  Or better yet, what if it's got varying authentication
>>    levels, some that impose a lighter admin/user burden?  Then we
>>    (admins) can just dial up whatever level we like, or turn it off
>>    completely.
>>
>>    Also, there's different ideas on how to design this.  But maybe
>>    that's no problem either.  We can be open to multiple solutions
>>    (see 4).
>>
>
> The question here, from the Metagovernment point of view, is: who gets to
> be the admin? The Metascore team's solution is to work toward a DVCS model
> where there is no singular admin in charge of a community, but it still
> seems like there is some basic software administration going on (or am I
> just not understanding it correctly?), and I still see problems with having
> a privileged user, as that opens a clear avenue for corruption.
>
> But as to the question of identity, it seems like the best solution is an
> external one. That should make standardization much easier. My vote here,
> however, would be to ensure that it is not a commercial one; again for fear
> of undue influence.
> Ed, what is DVCS model? What is external solution to identity? How would it
> help standardization? What is a commercial solution and how would it bring
> undue influence?
>

To me a unique identity is an uncompromising identity of a person and should
not be left to a chance or debated or voted  by any one.
