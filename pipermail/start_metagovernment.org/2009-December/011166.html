The more I think about this, there are a number of safety-features
which could be useful for protecting a wiki from undo damage caused by
a single proposal or the actions of a minority.

I would really like the 'watching' pages feature to be tied to
'tagging' -- one can essentially devote attention to the articles of
interest and ignore the rest.

But, more protection may be useful. Imagine, for example, that one
created a proposal called 'update FAQ' that, instead of updating the
FAQ actually deleted all pages. It's conceivable that a minority of
the community (5%) could sabotage the documents by passing this
proposal with little-to-no input from others.

Having a rapid-undo feature (a button next to 'history' item which
could quickly create a new proposal to undo all changes proposed by
previous proposal) could help recover from such a disaster. In
wikipedia's community, this kind of feature (roll-back or undo in
recent changes) is critically important to maintaining content.

Also, it might make sense to put a system summary (# articles changed,
# bytes added, # bytes removed, # new articles created, does this
change the Main Page) in the email to users notifying them that a
proposal is in the Voting phase.

These and some other suggestions I have entered into the Metagov
candiwi Suggested Features page, currently in Edit mode

(See http://metagov.candiwi.org/articles/from/Thoughts%2C%20Critiques%20and%20suggestions/at/305e4dc68a30/view/Suggested%20Features)

I will continue to add suggestions there, and encourage others to do the same.

Thanks,
Jacki
