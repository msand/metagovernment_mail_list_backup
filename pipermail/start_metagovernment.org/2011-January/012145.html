Ed and Alex,

Ed Pastore wrote:
> Perhaps this particular issue should be used as a use-case for
> crossforum architecture. We reached what appeared to be close to a
> consensus in Vilfredo (with one notable exception), but then in
> another forum (this list) we now see at least two people (a
> significant percentage) dissenting.
> 
> One notable thing that I see here is the importance of discussion,
> beyond mere voting. The context people have filled in both for and
> against proposals adds so much to my opinion of how I want to
> vote...

Maybe a general "gab feed" is needed?  I imagine it would work like
this:

 1. Aggregate all poll-related discussions
 2. Feed them into theatre app
 3. Project them onto map

It's similar to the diff feed in this regard, but it casts a wider net
and captures more posts.  For instance, here are the data for a single
post captured and formalized in a gab feed:

 a) Post URL:

    http://metagovernment.org/pipermail/start_metagovernment.org/2010-December/003304.html

 b) Short tweet-like summary of the post:

    The seemingly-endless Vilfredo question about the name of this
    thing we're doing reached the 16th generation.

 c) Poll ID - in this case it's reverse-mapped from
    http://vilfredo.org/viewquestion.php?q=53 :

    G/p/osd

 d) User ID:

    Epastore-MetagovernmentOrg

The feed might then proceed to capture all sub-posts that do not
explicitly alter the issue by reference to another.  Or maybe only
those that embed an actual reference.  (Not sure which is better.)

This will work with Vifredo-based discussions without requiring any
modification to Vilfredo.  It will also work with Votorola-based
discussions.  Others too, if we wanted to cast the net more widely.

Is this what's needed (at least at the moment)?

Alex wrote in "Pollserver Federation":
> It seems we are getting closer to a "standard" for the information
> needed for a "poll" or other consensus building process to continue
> online.

Working on something like a "gab feed" would give us an immediately
practical reason to document the underlying wiring as a
proto-standard.  I agree with Conseo, it's better to have a reference
implementation on the go when documenting this kind of stuff.  Both
the proto-standard and the implementation ought to be backed by an
actual proto-federation.
